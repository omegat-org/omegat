# Filling Pull Request Forms

This guide explains how to fill out the Pull Request (PR) form on GitHub. Proper documentation of your changes helps the OmegaT maintainers review and integrate your work more efficiently.

## 📝 PR Title

Your PR title should be a concise summary of the changes.

### Format
If your PR resolves a SourceForge ticket, please include the ticket number in square brackets at the beginning of the title.

- **With a ticket:** `[#123] fix: crash when opening project`
- **Without a ticket:** `docs: Update README with contribution guidelines`

## 🛠 Pull Request Type

Select the most appropriate type for your PR from the dropdown menu. This selection helps us categorize your contribution and automatically applies a corresponding label to the PR.

| Selection | Applied Label | Description |
| :--- | :--- | :--- |
| **Bug fix** | `bug` | Fixes an incorrect or unexpected behavior. |
| **Feature enhancement** | `enhancement` | Adds new functionality or improves existing features. |
| **Documentation** | `documentation` | Changes to manuals, guides, or other documentation. |
| **Build and release changes** | `build/release` | Changes to build scripts, dependencies, or release process. |
| **CI/CD changes** | `github_actions` | Changes to GitHub Actions or CI workflows. |
| **Dependency updates** | `dependencies` | Updates to project libraries and dependencies. |
| **Refactoring** | `refactoring` | Code changes that neither fix a bug nor add a feature. |
| **Localization/Translation** | `L10N/Translation` | Updates to UI translations or localized resources. |
| **UI/UX changes** | `UI` | Changes to the user interface or user experience. |
| **Other** | (None) | Use this if none of the above fit, and describe it in the "Other pull request type" field. |

## 🎫 SourceForge Ticket Information

OmegaT uses SourceForge for issue tracking. Even if you are submitting a PR on GitHub, it should ideally be linked to a SourceForge ticket.

1. **Ticket type:** Select whether it is a bug, feature request, or documentation ticket.
2. **Ticket number:** Enter the numeric ID of the ticket (e.g., `123`).
3. **Ticket title:** Paste the title of the ticket from SourceForge.

Providing this information allows the reviewer to easily find the background discussion and ensures the changelog is accurate.

## 📖 Description of Changes

In the **"What does this PR change?"** section, provide a bulleted list of the main changes. This is where you explain *what* you did and *why*.

> ✅ **Example:**
> - Added support for XYZ file format.
> - Fixed a null pointer exception in the ABC filter.
> - Updated the user manual to reflect the new settings.

## ℹ️ Other Information

Use the **"Other information"** section for anything else that might be helpful for the reviewer, such as:
- Screenshots of UI changes.
- Notes on how to test the changes.
- Any potential side effects or compatibility concerns.

## 🤖 Automated Labeling

When you submit or edit your PR, an automated workflow called **PR Labeler** runs in the background. It reads your selection in the "Pull request type" field and automatically adds the matching label to your PR. 

If you realize you selected the wrong type, you can edit the PR description, and the labels will be updated automatically.
